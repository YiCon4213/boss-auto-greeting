# 本地简历投递 Agent 产品需求

## 1. 产品目标

在现有 BOSS 直聘自动沟通油猴脚本基础上，增加本机 Agent、岗位方向分析、宽松筛选、个性化问候语、批量审批和可审计投递流程。

产品面向个人本地使用。它辅助用户处理本人有权查看的职位，不提供多用户、云端代投、账号共享或绕过平台限制的能力。

## 2. 双模式要求

### 2.1 独立模式

当本地 Agent 未启动或智能辅助关闭时，油猴脚本必须继续提供原有功能：

- 公司和 Boss 活跃度筛选。
- 跳过已沟通岗位。
- 固定常用语或自定义文本。
- 最短/最长发送间隔、等待时间和最大沟通数。
- 自动进入聊天、发送验证、返回列表、记录和导出。

### 2.2 Agent 模式

Agent 模式采用：

`自然语言建任务 -> 批次采集 -> 分析与生成 -> 人工审批 -> 不可变发送队列 -> 自动执行 -> 批次报告`

Agent 断开时，正在进行的智能批次必须暂停，不能悄悄降级成固定话术继续发送。

## 3. 自然语言 Agent

首版本地工作台提供简单对话入口，至少支持：

- 从当前 BOSS 列表创建指定大小的采集批次。
- 开关匹配分析和个性化话术。
- 调整批次大小、筛选阈值和发送间隔。
- 重新分析、重新生成某个岗位的话术。
- 打开画像、模型设置、审批和报告页面。
- 请求执行已经批准的批次。

自然语言输出不能直接转换为任意浏览器代码或数据库语句。Agent 只能调用预定义应用命令。没有批准批次时，“发送这些岗位”等自然语言只能打开审批页，不能直接发送。

## 4. 个人画像模板

### 4.1 求职目标

- 目标岗位。
- 技术方向。
- 希望从事的工作。
- 重点能力。
- 明确不考虑的方向。

### 4.2 个人情况

- 个人简介。
- 教育背景。
- 技能栈。
- 项目经历。
- 实习或工作经历。
- 科研经历。
- 竞赛经历。
- 开源项目。
- 优势亮点。
- 补充说明。

### 4.3 时间信息

- 可实习月份范围。
- 每周可到岗天数。
- 最早到岗时间。
- 其他时间约束。

### 4.4 字段规则

- 字段为空时不参与分析、评分或话术生成。
- 每个字段可以单独标记是否允许发送给模型。
- 电话、邮箱、证件号、详细住址等默认禁止发送给模型。
- 不得因个人画像未提及某项 JD 要求而判断用户不具备该能力。

## 5. 岗位分析与筛选

分析功能可以关闭。关闭后不计算匹配分、不自动取消岗位，但仍可独立启用个性化话术。

每个岗位生成：

- 求职目标匹配度，0 到 100。
- 个人情况匹配度，0 到 100。
- 参考综合分，仅用于排序。
- 主要匹配依据。
- 信息差距或风险提示。
- 建议状态：推荐、谨慎或方向偏离。

默认阈值：

- `0-29`：方向明显偏离，默认取消勾选。
- `30-59`：谨慎，默认保留。
- `60-100`：推荐，默认保留。

只有求职目标匹配度决定默认取消。个人情况匹配度不能单独淘汰岗位。地点和工作经验不参与方向筛选，因为用户可以在 BOSS 搜索条件中预先筛选。

所有默认取消的岗位仍保留在审批列表中，用户可以人工恢复。

## 6. 基础话术与生成规则

当前基础话术：

> 老师您好，我是来自华南农业大学（双一流）的研究生。仔细阅读贵公司的招聘要求，我认为自己能够胜任这份工作。本人具备AI Agent 和后端开发等技术栈和项目实践经历。在Github上有多个开源项目。此外还拥有丰富的科研，竞赛和工作经历，科研课题与机器学习，自然语言识别相关。可实习3个月至6个月，面试通过可3日内到岗

生成要求：

- 以基础话术为风格锚点，不重写成长求职信。
- 保留自然、直接、礼貌的语气和大致结构。
- 允许适度调整句式和长度，不设置机械字数限制。
- 每个岗位优先加入一至两个最相关匹配点。
- JD 强调技能时，匹配真实技能或项目。
- JD 强调实习周期和到岗时间时，匹配画像时间字段。
- JD 强调经验时，匹配工作、实习、项目、科研或竞赛经历。
- 没有可靠匹配点时保持基础模板，不强行个性化。
- 不能推测、夸大或虚构画像中不存在的信息。
- 用户人工修改后的终稿优先级最高，不再被模型覆盖。

个性化话术功能可以单独关闭。关闭后直接使用基础模板。

## 7. 批次与审批

- 默认每批采集 10 个岗位，数量可配置。
- 采集完成后停止并打开审批页，不自动继续下一批。
- 审批页采用简约左右分栏：左侧岗位队列，右侧详情、分析和话术编辑。
- 用户可以勾选、取消、恢复低分岗位、修改话术和重新生成。
- 一次点击批准本批次后，生成不可变发送队列。
- 只有已勾选且话术校验通过的岗位进入发送队列。

## 8. 刷新、重排与身份安全

审批基于持久化职位快照，不依赖旧 DOM。

发送每个岗位前必须：

1. 恢复对应搜索 URL 和求职期望。
2. 通过 `encryptJobId`、`securityId`、`lid` 等强标识重新定位岗位。
3. 打开当前详情并验证岗位身份。
4. 验证当前聊天会话属于获批岗位。
5. 检查是否已经沟通。
6. 通过后发送审批终稿。

同一岗位强 ID 下 JD 发生变化时，记录变化但继续使用审批话术。岗位下线、无法定位、已沟通或身份不一致时跳过当前岗位并继续批次。不得向身份无法确认的岗位发送。

## 9. 发送与异常策略

- 延续现有最短/最长随机间隔、等待上限和最大沟通数。
- 间隔用于合理限速和避免误操作，不以规避平台检测为目标。
- 单个岗位定位失败或发送失败时记录原因并继续。
- 批次结束后统一展示成功、跳过和失败报告。
- 失败岗位可由用户选择后重新执行，不在单次批次中无限重试。
- 遇到验证码、登录失效、安全校验或页面无法确认当前会话时立即暂停。

## 10. 模型接入

- 支持 OpenAI 兼容接口。
- 配置接口地址、模型名、API Key、超时和必要生成参数。
- 兼容云端服务和本地兼容服务。
- 分析和话术使用结构化响应契约。
- 无效 JSON、超时和字段缺失必须显式失败或有限重试。
- 模型失败后不得未经审批降级发送固定话术。

## 11. 本地工作台

首版保持简单，包含：

- Agent 对话。
- 当前批次。
- 审批工作台。
- 个人画像。
- 基础话术。
- 模型与系统设置。
- 投递记录和批次报告。

工作台由本地服务提供，只监听 `127.0.0.1`。

## 12. 数据与扩展性

- SQLite 长期保存画像、职位快照、分析、话术、审批、发送结果和去重信息。
- 支持按批次、按时间和全部清理。
- API Key 不保存到普通业务表、前端响应、日志或导出文件。
- 核心能力通过版本化 `/api/v1` 接口开放。
- 网页、内置自然语言 Agent 和未来上层项目调用同一应用服务。
- 首版不实现多用户、云端部署、多浏览器并发和复杂自治规划。

## 13. 验收标准

首版完成时必须能够：

1. 不启动 Agent 时继续运行原油猴独立模式。
2. 从自然语言创建一个当前列表的 10 岗位采集批次。
3. 保存完整职位快照并完成宽松双评分。
4. 根据基础模板生成自然的岗位话术。
5. 在简约分栏页面修改、勾选并批准批次。
6. 在列表重排后通过强 ID 重新定位获批岗位。
7. 逐条发送并验证结果，单条失败后继续。
8. 验证码或登录失效时暂停。
9. 在 SQLite 中查看完整批次报告。
10. 通过 `/api/v1` 从另一个本地项目创建、查询批次，但不能绕过审批发送。
