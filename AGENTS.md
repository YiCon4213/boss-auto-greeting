# 本地简历投递 Agent 开发约定

本仓库正在从单文件 BOSS 直聘自动沟通脚本，改造成一个双模式、本机运行、人工审批后发送、可被其他本地项目调用的简历投递 Agent。

本文是仓库级约束。直接用户指令优先于本文；已批准的产品需求和设计规格是实现依据。

## 开始任务前必读

按顺序阅读：

1. `AGENTS.md`
2. `docs/README.md`
3. `docs/current-state-analysis.md`
4. `docs/product-requirements.md`
5. `docs/superpowers/specs/2026-07-21-local-resume-delivery-agent-design.md`
6. `docs/roadmap.md`
7. 当前任务对应的详细实施计划（生成后位于 `docs/superpowers/plans/`）

修改前必须检查 Git 状态、最近提交和已有未提交改动，不覆盖或混入无关修改。

## 仓库和修改边界

- 只修改 `D:\Web\boss\boss-auto-greeting`。
- 不修改父目录 `D:\Web\boss` 原项目中的 Python 服务、Chrome 扩展、测试、启动脚本或文档。
- 本仓库是独立 Git 仓库，提交必须在本仓库中完成。
- 运行时数据库、密钥、日志、导出文件和本机配置不得提交到 Git。
- 功能、接口或安全边界改变时，同步更新产品需求、设计、README 和交接文档。

## 不可破坏的产品行为

### 独立模式

本地 Agent 未启动或智能辅助关闭时，`zhipin-auto-greeting.user.js` 必须继续独立提供：

- 公司、黑名单、Boss 活跃度和已沟通筛选。
- 固定常用语或自定义文本。
- 随机最短/最长间隔、等待上限、最大沟通数和聊天重试。
- 列表遍历、进入聊天、发送验证、返回列表、记录与导出。

后端、数据库、模型、构建工具或网络服务不得成为独立模式的启动依赖。

### Agent 模式

固定流程：

`自然语言建任务 -> 批次采集 -> 分析与生成 -> 人工审批 -> 不可变发送队列 -> 自动执行 -> 批次报告`

- 默认每批采集 10 个岗位，采集完成后停止并等待审批。
- 采集阶段只读取详情，不进入聊天、不发送消息。
- 没有人工批准的批次不得发送。
- Agent 断开时智能批次暂停，不能降级为固定话术继续发送。
- 单项普通失败记录后继续；验证码、登录失效和安全校验暂停整批。

## 总体架构边界

### 浏览器执行层

浏览器层负责读取 BOSS 列表、接口、详情和聊天页面，维护 SPA 路由与列表恢复，采集快照，重新定位已批准岗位，校验岗位与会话身份，发送并验证消息，回报任务结果。

浏览器层不得：

- 保存模型 API Key。
- 决定岗位是否值得投递。
- 修改已批准的最终话术。
- 执行后端返回的任意 JavaScript、选择器、SQL、URL 或未定义任务类型。

### 本地 Agent 服务

本地 FastAPI 服务负责：

- 个人画像、求职目标、基础话术和模型设置。
- SQLite 职位快照、分析、审批、发送状态和审计记录。
- OpenAI 兼容模型调用。
- 匹配分析和个性化话术生成。
- 批次状态机、审批门禁和发送队列。
- 本地网页工作台和版本化 `/api/v1`。
- 为未来上层项目提供安全的子 Agent 调用接口。

网页、内置自然语言 Agent 和未来外部项目必须调用同一应用服务，不能复制业务规则。

## 油猴脚本维护约束

- `zhipin-auto-greeting.user.js` 必须保持可直接安装运行。
- 首版不要在开发 Agent 功能的同时整体重写主脚本。
- 新增能力按职责接入：`StandaloneAutomation`、`AgentBridge`、`BatchCollector`、`ApprovedQueueRunner`。
- BOSS 是 SPA；列表与聊天可能通过 History API、完整导航或 BFCache 切换。
- 保留 `document-start` 注入，用于拦截 fetch/XHR、保存原生方法和捕获早期路由。
- 网络优先使用页面原生 fetch，仅在明确需要时使用 `GM_xmlhttpRequest`。
- 不扩大 `@connect` 权限，逐步将任意 `*` 收敛为明确允许的本地服务边界。
- 油猴面板保持原生 DOM 和内联样式，不要求引入 Vue、React 或 jQuery。
- 保持现有 IndexedDB/localStorage 兼容；新增 Agent 业务数据以 SQLite 为权威来源。
- 页面选择器、内部 API 和详情匹配变化必须带回归验证和诊断信息。

## 列表刷新、快照与身份校验

- 审批基于持久化快照，不能引用旧 DOM。
- 列表序号、滚动位置、岗位名称和公司名称不能单独作为发送身份。
- 使用 `encryptJobId`、`securityId`、`lid` 等强标识生成稳定身份。
- 发送前重新定位岗位、读取当前详情并确认聊天会话属于已批准岗位。
- 强 ID 相同但 JD 更新时，可以使用审批终稿继续发送并记录变化。
- 岗位下线、无法定位、已沟通或身份不一致时不得发送，记录后继续。
- 无法确认会话归属时停止当前项，禁止猜测成功。
- 只有聊天记录校验通过后才能写为 `sent`。

## 匹配分析规则

- 匹配分析可以独立关闭。
- 求职目标匹配度决定默认勾选：`0-29` 默认取消、`30-59` 谨慎、`60-100` 推荐。
- 个人情况匹配度只用于排序、解释和话术取材，不能淘汰岗位。
- 地点和工作经验不参与求职方向筛选。
- 画像没有的信息是“未知”，不能描述成用户不会、不具备或不满足。
- 默认取消的岗位仍显示在审批列表中，允许人工恢复。
- 分析关闭时所有岗位进入审批，不调用模型制造评分。

## 画像、模型与话术

- 画像模板包含求职目标、教育、技能、项目、工作/实习、科研、竞赛、开源和时间信息。
- 字段为空时不参与分析、评分或生成。
- 每个字段可以单独控制是否允许发送给模型。
- 电话、邮箱、证件号和详细住址默认禁止发送给模型。
- API Key 不得进入 SQLite 普通业务表、浏览器存储、前端响应、日志或导出。
- 模型使用 OpenAI 兼容协议；分析和话术使用独立结构化 schema。
- 无效 JSON、超时、字段缺失或最终失败必须显式记录；失败岗位不能进入发送队列。
- 基础话术是风格锚点，只做轻量个性化，不改写成长求职信。
- 优先匹配 JD 中的技能、实习周期、到岗时间或真实经历，通常加入一至两个有效匹配点。
- 不得推测、夸大或虚构个人经历。
- 人工终稿优先级最高，批准后不得重新生成或覆盖。

## API 与扩展性

- 对外应用接口统一使用 `/api/v1`。
- 本地网页、自然语言 Agent 和未来上层项目共享 Profile、Batch、Analysis、Greeting、Approval 和 Delivery 服务。
- 浏览器桥接接口与普通应用接口分开鉴权。
- 浏览器任务使用固定 schema、白名单类型、短期租约和幂等结果提交。
- 外部项目可以创建和查询批次、请求分析并取得审批 URL，但不能伪造批准状态或直接创建发送任务。
- 首版只监听 `127.0.0.1`，不做云端部署、多用户或多浏览器并发。
- 不把数据库、模型 SDK 或油猴实现细节泄漏给调用方。

## 安全与合规边界

- 仅处理用户本人登录后有权查看的职位。
- 不自动登录，不读取、导出或上传 Cookie、密码和浏览器凭据。
- 不绕过验证码、登录、安全校验、频率限制和访问控制。
- 随机间隔用于合理限速和减少误操作，不以规避检测为目标。
- `zhipin-devtools-unlock.user.js` 只作为隔离的本地调试辅助，不得成为正式运行依赖或继续扩展为反检测模块。
- 遇到验证码、登录失效、安全校验或站点明确阻断时立即暂停。
- 日志和导出排除完整个人画像、模型请求头、API Key、Cookie 和不必要的链接安全参数。

## 数据和状态

- SQLite 是 Agent 模式的长期业务数据源。
- 保存画像、基础话术、模型非敏感配置、批次、快照、分析、生成稿、人工终稿、审批版本、发送项和审计事件。
- 批准操作生成不可变审批版本和发送队列。
- 岗位身份、任务领取和结果提交必须幂等，防止重复快照、记录或发送。
- 数据库写入或审计状态无法保存时暂停批次，不能无记录发送。
- 提供按批次、按时间和全部清理。

## 开发与测试流程

- 先完成并审阅详细实施计划，再开始编码。
- 按阶段和测试驱动方式实施：先写失败测试，再做最小实现。
- 每个阶段都验证独立模式没有回归。
- 不做与目标无关的重构。
- 接口、状态机和持久化变更必须有自动化测试。
- DOM/SPA/聊天发送必须提供契约测试、可控夹具和人工浏览器验证清单。

最低测试范围：

- 画像空字段和敏感字段过滤。
- 默认 30 分筛选及个人分不淘汰。
- 模型结构化输出、失败和防虚构规则。
- 批次、审批和发送状态机的合法/非法转换。
- 未批准批次不能执行。
- 身份键、JD 指纹、去重和幂等。
- Agent 不在线时独立模式可用。
- Agent 采集阶段不发送。
- 列表刷新后通过强 ID 重新定位。
- 身份不一致时禁止发送。
- 单项失败继续，安全错误整批暂停。
- 外部子 Agent 不能绕过审批。

提交前必须运行与修改相称的完整验证命令，并检查输出、Git diff 和未提交文件。没有新鲜验证证据时，不得声称完成。

## 任务完成时的文档同步与清理

每个任务、阶段或修复完成后，提交前必须执行一次文档影响检查：

- 对照实际代码和验证结果，检查 `README.md`、`docs/README.md`、`docs/current-state-analysis.md`、`docs/roadmap.md`、当前阶段计划、手工验证清单和 `docs/new-session-handoff.md` 是否需要更新。
- 将当前计划中已经实际完成的复选框标记为完成，并记录精确测试数、覆盖率、人工验证状态和仍存在的非阻塞警告；不得提前勾选或伪造证据。
- 接口、端口、环境变量、数据表、迁移编号、状态名、启动命令或安全边界发生变化时，同步更新所有引用位置，并使用 `rg` 搜索旧名称和矛盾描述。
- `docs/README.md` 和 `docs/new-session-handoff.md` 是当前状态与新会话入口的权威索引；同一信息不要再创建按日期复制的交接文件。
- 删除文档前先确认它已经被更权威的文件完整替代、没有独特决策或验收证据、也没有仍在使用的引用。已批准规格、活动阶段计划、实施日志和人工验收记录不得仅因阶段完成而删除。
- 可以删除已被权威文档替代的重复交接、临时草稿、一次性提示词和没有后续用途的过程文件；删除后必须检查引用、`git diff --summary` 和 Git 状态，并在交付说明中列出删除内容及可恢复方式。
- 不提交运行时数据库、密钥、日志、导出、缓存、覆盖率产物或仅用于本机排障的临时文件。
- 最终验证必须包含文档一致性检查；若文档仍声称“尚未开始”“尚无测试”等与代码冲突的状态，不得结束任务。

## 代码规范

- 领域逻辑依赖接口，不直接依赖具体数据库、模型 SDK 或浏览器实现。
- 文件和模块保持单一职责，避免新的超大文件。
- 状态转换集中定义，不在 UI 和路由中复制判断。
- 使用清晰的数据模型和类型约束，边界输入必须校验。
- 区分可继续、单项失败、批次暂停和致命错误。
- 核心组件、兼容逻辑和高风险选择器要说明用途与失败语义。
- 注释重点解释身份、安全、恢复和兼容原因，不重复代码。
- 中文用户界面和文档保持清晰一致；内部标识和 API 字段使用稳定英文命名。
